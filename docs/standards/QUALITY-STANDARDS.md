# 품질 기준서 (Quality Standards)

> **목적**: 코드 리뷰/수정/개발 시 흔들리지 않는 절대 기준
> **작성일**: 2026-02-25
> **적용 범위**: 관리자페이지 전체 (backend, frontend, user-page, user-site)
> **원칙**: 이 문서가 모든 품질 판단의 최종 기준이다. 리뷰할 때마다 기준이 바뀌지 않는다.

---

## 1. 심각도 분류 (절대 기준)

코드 이슈는 반드시 아래 4등급으로만 분류한다. **주관적 판단 금지.**

| 등급 | 기준 | 수정 시점 | 예시 |
|------|------|-----------|------|
| **CRITICAL** | 금전 손실, 인증 우회, 데이터 손상이 **확실히** 발생 | 즉시 | 레이스컨디션으로 이중 지급, JWT 시크릿 하드코딩, 트랜잭션 밖 잔액 조작 |
| **HIGH** | 보안 취약점 또는 특정 조건에서 장애 발생 | 1주 내 | IDOR, N+1 쿼리(대량 데이터), 인증 누락 엔드포인트 |
| **MEDIUM** | 품질 저하이지만 당장 장애 아님 | 다음 스프린트 | 에러 핸들링 미비, 하드코딩 값, 불필요한 코드 |
| **LOW** | 코드 스타일, 네이밍, 미사용 import | 여유 시 | 변수명 개선, 주석 오타, 정렬 |

### 수정하지 않는 것들 (오버엔지니어링 방지)

- "이론적으로 가능한" 취약점 (현실적 공격 경로 없음)
- 성능 최적화 (실측 병목 확인 전)
- 코드 스타일 통일 (린터가 잡지 않는 것)
- "더 나은 패턴"으로의 리팩토링 (동작하면 두지 않는다)

## 2. 금전 관련 코드 절대 규칙

> 돈이 오가는 모든 코드에 적용. 위반 시 무조건 CRITICAL.

### 2-1. 트랜잭션 필수

```
잔액/포인트 변경 → 반드시 DB 트랜잭션 내부
검증 → 변경 → 로그 기록이 하나의 트랜잭션
```

### 2-2. 동시성 보호 필수

```
잔액 조회 후 변경 → SELECT FOR UPDATE 또는 원자적 UPDATE (SET balance = balance + amount)
Redis 카운터 → INCR (원자적) 먼저, 결과값으로 검증
```

### 2-3. 멱등성 필수

```
커미션/보상 지급 → 중복 방지 키 (reference_id + user_id + type UNIQUE)
DB 레벨 UNIQUE 제약 + 앱 레벨 사전 체크 (이중 방어)
```

### 2-4. 감사 추적 필수

```
모든 금전 변동 → MoneyLog 또는 PointLog 기록
자동 승인 → system_admin ID 사용 (None/null 금지)
balanceAfter 필드 → 실제 변경된 필드값 기록 (다른 필드값 기록 금지)
```

## 3. 인증/보안 절대 규칙

### 3-1. 시크릿 관리

```
JWT_SECRET, DB_PASSWORD → 프로덕션에서 환경변수 필수
하드코딩 폴백 → development 환경에서만 허용
production에서 미설정 시 → 프로세스 시작 거부 (throw/exit)
```

### 3-2. 토큰 생명주기

```
로그아웃 → access token + refresh token 둘 다 블랙리스트
토큰 갱신 → 이전 refresh token 블랙리스트 (Rotation)
갱신 실패 → 클라이언트에서 즉시 세션 정리 + 로그인 리다이렉트
```

### 3-3. 권한 검증

```
모든 API → 인증 미들웨어 통과 필수 (health 제외)
리소스 접근 → 소유자 검증 (user_id 매칭)
관리자 API → PermissionChecker 필수
```

## 4. 프론트엔드 절대 규칙

### 4-1. 상태 동기화

```
토큰 갱신 시 → localStorage + Zustand store 둘 다 업데이트
한쪽만 업데이트 → CRITICAL (UI와 실제 인증 상태 불일치)
```

### 4-2. 입력 검증

```
URL 파라미터(id 등) → Number 변환 후 isNaN 체크 필수
사용자 입력 → Zod 또는 동등한 런타임 검증
API 응답 → 필수 필드 존재 확인 후 렌더링
```

### 4-3. 에러 처리

```
401 응답 → 토큰 갱신 시도 → 실패 시 로그인 리다이렉트
네트워크 에러 → 사용자에게 안내 (무한 로딩 금지)
빈 데이터 → 빈 상태 UI 표시 (크래시 금지)
```

## 5. 코드 리뷰 프로토콜

### 5-1. 리뷰 범위 제한 (무한 루프 방지)

```
1회 리뷰 → CRITICAL + HIGH만 수정
MEDIUM/LOW → 목록 기록만, 즉시 수정 안 함
수정 후 재리뷰 → 수정한 부분만 확인 (전체 재스캔 금지)
```

### 5-2. 리뷰 완료 기준

```
CRITICAL 0건 → 리뷰 통과
HIGH가 있어도 CRITICAL 0이면 → 배포 가능 (HIGH는 후속 처리)
```

### 5-3. 리뷰-수정 사이클 상한

```
최대 2회 사이클 (리뷰→수정→확인리뷰→완료)
3회 이상 반복 → 아키텍처 문제. 수정이 아니라 재설계 필요.
```

## 6. 새 기능 개발 시 필수 체크리스트

### 금전 관련 기능

- [ ] 트랜잭션 내부에서 잔액 변경?
- [ ] SELECT FOR UPDATE 또는 원자적 UPDATE 사용?
- [ ] 중복 방지 키 UNIQUE 제약?
- [ ] MoneyLog/PointLog 정확한 필드 기록?
- [ ] 자동 처리 시 system admin ID 사용?

### 인증 관련 기능

- [ ] 인증 미들웨어 적용?
- [ ] 리소스 소유자 검증?
- [ ] 시크릿 환경변수 필수 설정?

### 프론트엔드

- [ ] URL 파라미터 NaN 가드?
- [ ] 토큰 갱신 시 store 동기화?
- [ ] 401 처리 → 로그인 리다이렉트?
- [ ] 빈 데이터/에러 상태 UI?

## 7. 이 문서의 사용법

```
1. 새 코드 작성 → 해당 섹션 체크리스트 확인
2. 코드 리뷰 → 심각도 분류표로 이슈 등급 지정
3. 수정 결정 → CRITICAL만 즉시, 나머지는 기록
4. 리뷰 완료 판단 → CRITICAL 0건이면 통과
5. 반복 리뷰 → 최대 2회, 초과 시 재설계 논의
```

**이 기준은 바뀌지 않는다.** 새 패턴이 필요하면 이 문서에 추가한다.
